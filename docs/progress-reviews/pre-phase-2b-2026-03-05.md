## Scalability Concerns

| Issue | Impact at Scale | Current State | Recommendation |
|-------|-----------------|---------------|----------------|
| **Missing index: `shop_reviews(shop_id)`** | Full table scan on every shop detail page; N+1 at scale | No index defined in migrations | Add migration: `CREATE INDEX idx_shop_reviews_shop ON shop_reviews(shop_id)` |
| **Missing index: `shops(processing_status)`** | Pipeline state queries (find pending shops) full-scan shops table | No index | Add migration: `CREATE INDEX idx_shops_processing_status ON shops(processing_status)` |
| **Missing index: `profiles(deletion_requested_at)`** | Daily hard-delete scheduler scans all profiles | No index | Add migration: `CREATE INDEX idx_profiles_deletion_requested ON profiles(deletion_requested_at) WHERE deletion_requested_at IS NOT NULL` |
| **Missing index: `shops(source)`** | Analytics/filtering by source (cafe_nomad vs user_submission) full-scans | No index | Add migration: `CREATE INDEX idx_shops_source ON shops(source)` |
| **`confirmed_tags` is unvalidated free array** | Users can submit arbitrary tag IDs not in taxonomy | CHECK constraint not enforced | Either add FK-like validation in Python or add DB trigger |
| **`search_shops` RPC loads all photos/tags per result** | Subqueries per result row; acceptable at 20 results, risky at 50+ | COALESCE subqueries inside RPC | Acceptable for MVP; add caching layer if load justifies |

## Key Insights

1. **DB-level enforcement is the real safety net.** The 3-list cap, photo requirement, and stamp creation are all enforced at the DB trigger level — meaning even if the Python API has a bug, the data stays consistent. This is excellent defensive architecture.

2. **The provider abstraction pattern is consistently applied.** After reviewing all 50 backend test files and all 6 provider adapters, there is not a single SDK import leak into business logic. This means swapping any external provider (OpenAI → Google for embeddings, Resend → SendGrid for email) requires changing only the adapter file and the factory function.

3. **Frontend test gaps cluster around auth entry points.** The core product flows (check-in, lists, profile) are well-tested. The gaps are specifically in the entry points (login, signup) and the PDPA compliance flow (consent, recovery). This is the classic "test the happy path, skip the onboarding" antipattern — fixable with targeted test additions.
